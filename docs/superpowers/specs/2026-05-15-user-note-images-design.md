# 备注图片支持设计

## 背景

当前知识库详情页中的 `我的备注` 仅支持纯文本备注，复习卡背面也只能展示文本内容。需求是让用户可以在备注输入框中直接粘贴多张图片并保存到后端本地文件夹中，随后在：

- 知识库详情页中以较大尺寸查看备注图片，并支持点开大图
- 复习卡背面的备注区以小尺寸缩略图展示，并支持点开大图
- 备注与图片一起支持新增、编辑、删除

本次设计只覆盖单词/知识库详情页的 `我的备注` 以及复习页对该备注的读取与展示，不改动“快速记录”主笔记创建流程。

## 目标

1. 支持用户在备注编辑框中粘贴多张图片并保存。
2. 图片文件落地到后端本地目录，数据库只保存相对访问路径。
3. 单条备注支持三种内容形态：纯文字、纯图片、文字加多图。
4. 详情页支持备注的新增、编辑、删除和大图预览。
5. 复习卡背面在备注区顶部显示紧凑文字和小图缩略图，并支持大图预览。

## 非目标

1. 不为主 `Note` 模型新增图片字段。
2. 不做云存储、对象存储或 CDN。
3. 不做图片压缩、裁剪、拖拽排序、批量下载。
4. 不做跨备注复用图片。
5. 不做备注图片的独立搜索、标签或封面选择。

## 当前状态

### 后端

- `NoteUserNote` 目前只有 `content` 字段，没有图片字段。
- 备注接口仅支持：
  - `GET /notes/:id/user-notes`
  - `POST /notes/:id/user-notes`
  - `DELETE /notes/:id/user-notes/:userNoteId`
- 上传能力在项目中已存在基础用法，`import` 模块已使用 `FileInterceptor`。
- `ServeStaticModule` 已用于暴露写作资源目录，可复用相同模式暴露备注图片目录。

### 前端

- `KnowledgeDetail` 页面的 `我的备注` 当前为纯文本列表。
- 备注只支持新增文本和删除，不支持编辑。
- `ReviewCards` 通过 `GET /notes/:id/user-notes` 拉取文本备注，并在卡片背面顶部显示。
- 当前 `UserNote` 前端类型只有 `id` 和 `content`。

## 方案概览

采用“在现有备注对象上直接扩展图片数组”的方案，而不是新增图片子表。

每条备注仍然是一条 `NoteUserNote` 记录，但其内容从“文本”扩展为“文本 + 图片数组”：

- 文本和图片属于同一条备注
- 图片与文字一起新增、一起编辑、一起删除
- 后端文件系统保存实际图片
- 数据库只保存图片相对访问路径数组

该方案能最小化改动范围，同时满足多图、编辑、详情展示和复习展示的需求。

## 数据模型设计

### Prisma 模型调整

`NoteUserNote` 调整为：

- `content: String @default("")`
- `images: String[] @default([])`
- 保留 `id`、`noteId`、`createdAt`、`updatedAt`、`deletedAt`

设计理由：

1. `content` 保持字符串而非可空，避免前后端大量 `null` 分支。
2. 纯图片备注使用空字符串 `""` 表示无文本内容。
3. `images` 直接使用 `String[]`，适合当前“只需保存 URL/path 列表”的需求，比 `Json` 更简单。

### 返回结构

后端返回的备注对象统一为：

```ts
type UserNote = {
  id: string
  content: string
  images: string[]
  createdAt: string
  updatedAt: string
}
```

前端据此渲染文本和缩略图，不再将备注视为单纯字符串。

## 文件存储设计

### 存储位置

图片文件保存在后端本地目录：

`backend/uploads/note-user-images/`

可按年月分目录，例如：

`backend/uploads/note-user-images/2026/05/<uuid>.<ext>`

### 访问路径

后端通过静态资源暴露：

- 本地目录：`backend/uploads/note-user-images`
- 访问前缀：`/note-user-images`

数据库中保存对前端可直接访问的相对路径，例如：

- `/note-user-images/2026/05/abc123.webp`

### 文件命名

保存时统一重命名为随机文件名，避免冲突和路径注入：

- 文件名不保留用户原名
- 扩展名保留为源图片格式对应扩展名

### 删除策略

1. 删除整条备注时，删除该备注关联的全部图片文件。
2. 编辑备注时，如果用户移除了某些旧图，则保存成功后删除这些图片文件。
3. 若文件已不存在，删除操作应容忍失败并继续主流程，不影响接口成功返回。

## 后端接口设计

## GET /notes/:id/user-notes

用途：

- 拉取某条知识库笔记下的全部备注及图片

返回：

```ts
{
  items: Array<{
    id: string
    content: string
    images: string[]
    createdAt: string
    updatedAt: string
  }>
}
```

## POST /notes/:id/user-notes

用途：

- 新建备注

请求格式：

- `multipart/form-data`

字段：

- `content`: string
- `images`: File[]，支持多张

校验规则：

1. `content.trim()` 为空且 `images.length === 0` 时拒绝保存。
2. 单条备注最多 10 张图片。
3. 单张图片大小上限 5MB。
4. 仅允许 `image/png`、`image/jpeg`、`image/webp`、`image/gif`。

返回：

- 返回完整的 `UserNote` 对象，前端可直接插入列表。

## PATCH /notes/:id/user-notes/:userNoteId

用途：

- 编辑备注文字
- 删除部分旧图
- 追加新图

请求格式：

- `multipart/form-data`

字段：

- `content`: string
- `keepImages`: JSON 字符串数组，表示本次保存后要保留的旧图路径
- `images`: File[]，表示本次新增上传的图片

示例：

```json
["/note-user-images/2026/05/a.webp", "/note-user-images/2026/05/b.png"]
```

保存逻辑：

1. 查询该备注当前旧图列表。
2. 解析 `keepImages`，只允许保留属于该备注的旧图路径。
3. 上传新增图片并生成新路径。
4. 最终保存的 `images = keepImages + newUploadedImages`。
5. 将“旧图列表 - keepImages”得到的图片文件从磁盘删除。
6. 若最终 `content.trim()` 为空且 `images.length === 0`，则拒绝保存。

返回：

- 返回更新后的完整 `UserNote` 对象。

## DELETE /notes/:id/user-notes/:userNoteId

用途：

- 删除备注

行为：

1. 软删除备注记录。
2. 删除该备注关联的全部图片文件。

返回：

- 与现有删除接口风格保持一致，不要求返回复杂数据。

## 后端实现拆分

### 控制器

`NotesController` 需要新增：

- `PATCH :id/user-notes/:userNoteId`
- 备注创建接口从纯 JSON 改为 `multipart/form-data`
- 创建和编辑接口都使用多文件拦截器

### 服务层

`NotesService` 需要新增能力：

1. 校验备注归属和未删除状态
2. 规范化文本内容
3. 保存上传文件并生成对外路径
4. 合并旧图与新图
5. 删除被移除的图片文件
6. 删除整条备注时清理全部图片

建议将文件系统相关逻辑提取为内部私有辅助函数，避免控制器/服务主逻辑过长。

### 模块与静态资源

`AppModule` 中追加一条 `ServeStaticModule.forRoot(...)` 用于暴露备注图片目录，保持与已有写作资源静态服务一致的模式。

## 前端设计

## 类型调整

前端 `UserNote` 改为：

```ts
type UserNote = {
  id: string
  content: string
  images: string[]
  createdAt?: string
  updatedAt?: string
}
```

`KnowledgeDetail` 和 `ReviewCards` 不再把备注视为 `string[]`，而是结构化对象数组。

## 知识库详情页

### 展示形态

`我的备注` 区块升级为“备注卡片列表”：

- 每条备注单独成块
- 上方为正文文本（如有）
- 下方为图片缩略图网格（如有）
- 保留删除按钮
- 新增编辑按钮

支持的备注内容：

1. 纯文字
2. 纯图片
3. 文字 + 多图

### 新增备注交互

点击“添加备注”后，展开编辑器：

1. 一个文本框用于输入备注内容
2. 一个图片区用于显示已粘贴图片预览
3. 保存 / 取消按钮

用户在文本框聚焦状态下粘贴图片时：

1. 拦截 `paste` 事件
2. 从 `clipboardData.items` 中提取图片文件
3. 将图片加入本地待上传列表
4. 生成本地预览 URL
5. 在图片区显示缩略图

每张待上传图片支持：

- 查看预览
- 单独移除

保存时：

- 使用 `FormData`
- 传入 `content`
- 多张图片以同名字段 `images` 追加

### 编辑备注交互

每条备注增加“编辑”按钮，进入内联编辑态：

1. 文本框回填原内容
2. 原图片以“已保存图片”形式展示
3. 支持从已保存图片中移除部分图片
4. 支持继续粘贴新增图片
5. 点击保存后提交 `content + keepImages + new images`

取消编辑时：

- 丢弃本地新增图片
- 恢复到服务端最近一次保存状态

### 图片展示

详情页中的备注图片使用较大缩略图：

- 建议边长 80 到 120 像素
- 支持多图换行网格展示

点击任意图片后打开大图预览层：

- 遮罩层覆盖当前页面
- 展示当前图片大图
- 同一条备注内支持左右切换
- 点击遮罩、关闭按钮或按 `Escape` 关闭

## 复习卡背面

### 展示位置

备注区继续保持在卡片背面的靠上位置，优先于其他解释性内容出现。

### 展示形态

复习页仅做轻量展示，不提供编辑能力：

- 文字显示 1 到 2 行摘要
- 图片显示小尺寸缩略图，建议 32 到 40 像素
- 图片较多时只展示前几张，末尾显示 `+N`

### 交互

点击任意缩略图后：

- 打开与详情页同风格的大图预览层
- 不跳转页面
- 关闭后仍留在当前复习卡

### 数据读取

复习页获取 `userNotes` 时不再只取 `content`，而是取完整对象，以便展示图片。

## 图片预览组件

建议前端抽一个可复用图片预览组件，用于：

1. 详情页备注区
2. 复习卡背面

组件职责：

- 渲染缩略图
- 打开/关闭大图预览
- 在同一组图片中左右切换

这样可以避免详情页和复习页维护两套不同的大图逻辑。

## 校验与错误处理

### 前端校验

在提交前做第一层校验：

1. 文本为空且图片为空时，不允许保存。
2. 超过 10 张图时阻止继续添加。
3. 非图片内容粘贴时按普通文本处理，不报错。
4. 单张超 5MB 或格式不支持时立即提示，并保留其他合法图片。

### 后端校验

后端必须重复校验，不信任前端：

1. 文本和图片不能同时为空
2. 图片数量上限
3. MIME 类型白名单
4. 文件大小上限
5. `keepImages` 只能包含属于当前备注的旧图路径

### 失败行为

1. 保存失败时，不清空当前编辑态文本和待上传图片。
2. 编辑失败时，页面保持编辑态，允许用户继续重试。
3. 如果新图片已写入磁盘但数据库保存失败，应对本次新写入文件做最佳努力回滚删除，避免产生孤儿文件。
4. 删除图片文件失败不应导致主记录操作回滚，但应记录日志。

## 性能与约束

1. 首版不做图片压缩和转码，降低实现复杂度。
2. 首版不做虚拟滚动或懒加载，备注区图片规模受数量限制控制。
3. 通过“每条备注最多 10 张图、每张最多 5MB”控制体量。
4. 复习页只展示小缩略图，避免背面内容过重。

## 风险与应对

### 风险 1：编辑接口协议复杂

风险：

- 编辑时既要保留旧图，又要上传新图，还要删除被移除旧图

应对：

- 统一使用 `keepImages + images[]` 协议，避免“增删改图片”出现多个相互冲突的字段

### 风险 2：文件与数据库状态不一致

风险：

- 数据库更新成功但文件删除失败，或上传成功但后续保存失败

应对：

- 保存新文件后再更新数据库
- 如果数据库保存失败，立即对本次新增文件做最佳努力清理
- 文件清理失败只记日志，不阻断主流程
- 接口主成功条件以数据库状态为准

### 风险 3：复习卡背面空间过小

风险：

- 多备注、多图可能挤占学习内容空间

应对：

- 复习页采用小缩略图和文本摘要
- 图片多时折叠为前几张加 `+N`

## 开放决策

本次设计已固定以下决策，不再留作实现时临时决定：

1. 使用 `NoteUserNote.images: String[]`
2. 图片文件保存在后端本地目录
3. 详情页支持备注新增、编辑、删除
4. 复习页只读展示，不支持编辑
5. 首版限制每条备注最多 10 张图、单图最大 5MB
